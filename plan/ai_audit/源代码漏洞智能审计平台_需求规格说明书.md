# Argus 源代码漏洞智能审计平台需求规格说明书

> 文档版本：v4.0
> 更新日期：2026-04-27
> 适用项目：Argus 当前代码库
> 状态：项目对齐版需求规格

## 1. 文档目的与边界

### 1.1 文档目的

本文档用于指导 Argus 当前项目中的“智能审计”能力建设。文档以现有代码库为事实基础，约束智能审计的产品流程、后端接口、前端页面、数据模型、运行时边界、验收标准与后续演进方向。

本文档不是独立新系统设计，不要求重建一套与 Argus 脱离的智能体平台。所有需求均应优先复用当前 Argus 的 React 前端、Rust 后端、任务状态快照、Hermes 运行时、Prompt Skill 管理、LLM 配置、Opengrep 静态审计、统一漏洞详情和报告导出能力。

### 1.2 当前项目事实

| 项目面 | 当前实现 |
| --- | --- |
| 前端框架 | React + Vite + TypeScript |
| 后端框架 | Rust + Axum |
| 数据持久化 | Postgres 用于业务配置与项目数据；任务运行态主要由 `rust-task-state.json` 快照保存 |
| 智能审计入口 | `/tasks/intelligent` 创建与管理任务，`/agent-audit/:taskId` 查看任务详情 |
| 静态审计入口 | `/tasks/static`、`/static-analysis/:taskId`、Opengrep 规则管理 |
| 智能审计后端接口 | `/api/v1/agent-tasks/*` |
| 智能体运行时 | Hermes manifests + `docker exec hermes chat` 交接 |
| 当前 Hermes 角色 | `recon`、`analysis`、`verification`、`report` |
| LLM 配置 | `/api/v1/system-config/*`，支持 provider、model、baseUrl、apiKey、customHeaders、连接测试与预检 |
| Prompt Skill | `/api/v1/skills/*`，支持 builtin prompt、custom prompt、agent-specific prompt 与 runtime snapshot |
| 漏洞详情 | `/finding-detail/:source/:taskId/:findingId` 统一详情页，兼容智能审计与静态审计 |
| 部署方式 | `docker compose up --build`，包含 frontend、backend、db、redis、opengrep-runner 等服务 |

### 1.3 已废弃的旧文档假设

旧版文档中的下列设计不属于当前 Argus 的已实现架构，不能作为当前需求直接落地：

- 独立的 AgentFlow DSL 编排层。
- Scratchboard 共享日志数据库。
- HarnessOpt 自动演化智能体拓扑。
- ECS / EC2 / SSH 远程执行调度。
- 默认 GPU 本地推理。
- 动态专家容器池自动扩缩容。
- 独立于 Argus 后端之外的新编排后端。

这些概念只能作为远期研究方向，不能写入 P0/P1 当前实施范围。

## 2. 产品目标

### 2.1 总体目标

智能审计用于在已有项目代码上执行 AI Agent 辅助的漏洞分析。平台应让用户能够选择项目、配置 LLM、启动智能审计任务、观察任务过程、查看实时发现、复核漏洞证据、标记误报或确报，并导出任务级或漏洞级报告。

### 3.3 当前部署边界

当前部署以本地 Docker Compose 为主：

- `frontend`：Vite 开发服务，对外端口默认 `13000`。
- `backend`：Rust 后端，对外端口默认 `18000`。
- `db`：Postgres。
- `redis`：Redis，供后续缓存和运行态能力使用。
- `opengrep-runner`：用于构建和预热本地 Opengrep runner 镜像；真实扫描由 backend 动态拉起 runner 容器。
- backend 挂载 Docker socket，用于 runner preflight 和动态容器执行。
- backend 挂载 `./backend/agents` 到容器内 `/app/backend/agents`，作为 Hermes manifest 发现目录。

## 4. 智能审计业务流程

### 4.1 任务创建流程

1. 用户进入 `/tasks/intelligent`。
2. 用户点击“新建扫描任务”。
3. 创建弹窗加载项目列表，只允许选择有效 Zip 项目；也可在入口允许时上传新项目。
4. 当模式为智能审计时，前端调用 agent preflight：
   - 检查 LLM provider。
   - 检查 model。
   - 检查 baseUrl。
   - 非 Ollama provider 检查 apiKey。
   - 必要时展示快速配置面板，要求保存并手动测试成功后才能创建。
5. 用户可填写目标文件范围，形成 `target_files`。
6. 前端调用 `createAgentTask`，默认设置：
   - `name`: `智能审计-{project.name}`
   - `description`: 智能审计任务标记
   - `use_prompt_skills`: `true`
   - `verification_level`: `analysis_with_poc_plan`
   - `target_files`: 用户输入的文件范围，可为空
7. 后端创建 `AgentTaskRecord`，状态为 `pending`，写入任务快照。
8. 前端可导航到 `/agent-audit/:taskId` 查看任务详情。

### 4.2 任务启动流程

1. 前端或任务详情页调用 `POST /api/v1/agent-tasks/{task_id}/start`。
2. 后端读取任务快照，设置 `started_at` 和最小迭代数。
3. 后端尝试 Hermes dispatch：
   - 从 `HERMES_AGENTS_BASE_PATH` 发现 agent manifest。
   - 按 `recon`、`analysis`、`verification`、`report` 顺序交接。
   - 每个角色调用对应容器的 `hermes chat -Q -q <payload> --source tool`。
   - 执行结果写入 agent tree。
4. 如果 Hermes manifests 不存在或目录不可用，后端走当前 mock completed 降级路径。
5. 如果 Hermes 部分失败，任务状态为 failed，并在错误信息和事件中暴露失败原因。

### 4.3 任务观察流程

任务详情页应同时支持实时流和补偿轮询：

- SSE 实时事件：`GET /api/v1/agent-tasks/{task_id}/events`。
- 事件列表补偿：`GET /api/v1/agent-tasks/{task_id}/events/list?after_sequence=...`。
- 漏洞补偿刷新：`GET /api/v1/agent-tasks/{task_id}/findings`。
- 任务详情刷新：`GET /api/v1/agent-tasks/{task_id}`。
- Agent 树：`GET /api/v1/agent-tasks/{task_id}/agent-tree`。
- Checkpoints：`GET /api/v1/agent-tasks/{task_id}/checkpoints`。

### 4.4 漏洞复核流程

1. 任务详情页展示实时漏洞列表。
2. 用户打开漏洞详情，查看：
   - 漏洞标题和类型。
   - CWE 编号与名称。
   - 影响文件和行号。
   - 代码片段和上下文。
   - 可达性证据。
   - 触发链或调用链。
   - PoC 或 PoC 计划。
   - 验证证据。
   - 修复建议。
3. 用户可将漏洞标记为：
   - `open`：待验证。
   - `verified`：确报。
   - `false_positive`：误报。
4. 后端更新 finding 状态，并重新计算任务统计和缺陷摘要。
5. 项目详情页和仪表盘应使用同一统计口径展示智能审计结果。

### 4.5 报告导出流程

任务级报告：

- `GET /api/v1/agent-tasks/{task_id}/report?format=markdown|json|pdf`
- 支持参数：
  - `include_code_snippets`
  - `include_remediation`
  - `include_metadata`
  - `compact_mode`

漏洞级报告：

- `GET /api/v1/agent-tasks/{task_id}/findings/{finding_id}/report?format=markdown|json|pdf`
- 文件名应包含项目名和漏洞标题的安全化片段。

## 5. 数据模型需求

### 5.1 AgentTaskRecord

智能审计任务必须至少包含以下信息：

| 字段类别 | 字段 |
| --- | --- |
| 身份 | `id`、`project_id`、`name`、`description`、`task_type` |
| 状态 | `status`、`current_phase`、`current_step`、`error_message` |
| 时间 | `created_at`、`started_at`、`completed_at` |
| 进度 | `progress_percentage`、`total_iterations` |
| 范围 | `audit_scope`、`target_vulnerabilities`、`verification_level`、`exclude_patterns`、`target_files` |
| 统计 | `total_files`、`indexed_files`、`analyzed_files`、`files_with_findings`、`findings_count`、`verified_count`、`false_positive_count` |
| 严重程度 | `critical_count`、`high_count`、`medium_count`、`low_count`、`verified_*_count` |
| Agent 指标 | `tool_calls_count`、`tokens_used`、`quality_score`、`security_score` |
| 结果 | `report`、`events`、`findings`、`checkpoints`、`agent_tree` |

### 5.2 AgentFindingRecord

漏洞记录必须覆盖“可复核”需求，而不只是模型结论。

| 字段类别 | 字段 |
| --- | --- |
| 身份 | `id`、`task_id` |
| 类型 | `vulnerability_type`、`cwe_id`、`cwe_name` |
| 展示 | `title`、`display_title`、`description`、`description_markdown` |
| 位置 | `file_path`、`line_start`、`line_end`、`resolved_file_path`、`resolved_line_start` |
| 代码 | `code_snippet`、`code_context`、`context_start_line`、`context_end_line` |
| 状态 | `status`、`is_verified`、`verdict`、`authenticity` |
| 可达性 | `reachability`、`reachability_file`、`reachability_function`、`flow_path_score`、`flow_call_chain` |
| 触发链 | `function_trigger_flow`、`trigger_flow`、`poc_trigger_chain`、`flow_control_conditions` |
| 业务逻辑证据 | `logic_authz_evidence` |
| PoC | `has_poc`、`poc_code` |
| 修复 | `suggestion`、`fix_code` |
| 报告与解释 | `report`、`ai_explanation`、`ai_confidence`、`confidence` |
| 时间 | `created_at` |

### 5.3 AgentEventRecord

事件用于实时观察和审计追踪：

- `event_type`：事件类型，如 task_start、tool、finding、phase、error。
- `phase`：当前阶段。
- `message`：中文可读消息。
- `tool_name`、`tool_input`、`tool_output`、`tool_duration_ms`：工具调用证据。
- `finding_id`：关联漏洞。
- `tokens_used`：令牌消耗。
- `metadata`：扩展信息。
- `sequence`：递增序列，供 SSE 断线续传和事件补偿。
- `timestamp`：事件时间。

### 5.4 AgentCheckpointRecord

Checkpoint 用于保存阶段性状态：

- Agent 身份：`agent_id`、`agent_name`、`agent_type`、`parent_agent_id`。
- 执行情况：`iteration`、`status`、`total_tokens`、`tool_calls`、`findings_count`。
- 内容：`checkpoint_type`、`checkpoint_name`、`state_data`、`metadata`。

### 5.5 缺陷摘要统计

智能审计任务应输出缺陷摘要：

```json
{
  "scope": "all_findings",
  "total_count": 0,
  "severity_counts": {
    "critical": 0,
    "high": 0,
    "medium": 0,
    "low": 0,
    "info": 0
  },
  "status_counts": {
    "pending": 0,
    "verified": 0,
    "false_positive": 0
  }
}
```

### 6.2 创建任务请求

```json
{
  "project_id": "uuid",
  "name": "智能审计-项目名",
  "description": "智能审计任务",
  "audit_scope": {},
  "target_vulnerabilities": [
    "sql_injection",
    "xss",
    "command_injection",
    "path_traversal",
    "ssrf"
  ],
  "verification_level": "analysis_with_poc_plan",

  "max_iterations": 8,
  "use_prompt_skills": true

```

### 6.3 查询漏洞参数

`GET /findings` 必须支持：

- `severity`
- `vulnerability_type`
- `verified_only`
- `include_false_positive`
- `skip`
- `limit`

列表应按严重程度、确认状态、创建时间等稳定顺序返回。

### 6.4 状态更新规则

更新漏洞状态后，后端必须重新计算：

- `findings_count`
- `verified_count`
- `false_positive_count`
- 严重程度计数
- 已验证严重程度计数
- `defect_summary`
- `files_with_findings`

误报不应计入有效漏洞数，但仍应保留在任务历史中。

## 7. 前端需求

### 7.1 首页入口

首页提供两个主入口：

- 静态审计：规则驱动漏洞检测。
- 智能审计：AI Agent 代码推理。

智能审计入口跳转到 `/tasks/intelligent?openCreate=1&source=home-agent`。

### 7.2 智能任务管理页

页面路径：`/tasks/intelligent`

必须提供：

- 任务状态统计卡片：总任务、已完成、进行中。
- 项目名搜索。
- 新建扫描任务按钮。
- 任务列表表格。
- 对 pending/running 任务进行定时刷新。
- 创建成功后刷新列表。

### 7.3 创建智能审计弹窗

必须提供：

- 项目选择，默认仅展示可扫描 Zip 项目。
- 可选上传项目。
- 目标文件输入。
- LLM 配置预检。
- 快速修复面板：
  - provider。
  - model。
  - baseUrl。
  - apiKey。
  - 保存配置。
  - 手动测试连接。

### 7.4 智能任务详情页

页面路径：`/agent-audit/:taskId`

必须提供：

- 顶部状态、返回、导出。
- 任务统计：进度、耗时、漏洞数、有效漏洞、误报、迭代、工具调用、token。
- 事件日志：
  - 支持 SSE。
  - 支持断线恢复和事件列表补偿。
  - 支持工具输入输出详情。
- Agent 树：
  - root。
  - recon。
  - analysis。
  - verification。
  - report。
- 实时漏洞列表：
  - 分页。
  - 严重程度过滤。
  - 关键字过滤。
  - 状态标签。
- 漏洞详情弹窗或统一详情页跳转。
- 任务级报告导出。

### 7.5 统一漏洞详情页

页面路径：`/finding-detail/:source/:taskId/:findingId`

必须兼容：

- `source=agent`
- `source=static`

智能审计漏洞详情必须展示：

- 漏洞类型和 CWE。
- 危害等级和置信度。
- 文件位置。
- 代码上下文。
- AI 分析解释。
- 可达性和触发链。
- PoC 或 PoC 计划。
- 验证证据。
- 修复建议。
- 单漏洞报告下载。

### 7.6 项目详情页联动

项目详情页应把智能审计纳入项目维度视图：

- 最近任务包含智能审计任务。
- 潜在漏洞统计包含智能审计确报和待验证漏洞。
- “查看漏洞详情”进入同一套漏洞详情体验。
- 智能审计漏洞类型展示应使用 CWE 或清晰的漏洞类型标签。

## 8. LLM 与 Prompt Skill 需求

### 8.1 LLM 配置

系统必须支持：

- provider catalog。
- provider alias 规范化。
- 默认 baseUrl 和默认 model。
- apiKey 字段映射。
- customHeaders 校验。
- 手动连接测试。
- agent preflight。

预检失败时必须返回：

- `ok`
- `stage`
- `message`
- `reason_code`
- `missing_fields`
- `effective_config`
- `saved_config`

### 8.2 Prompt Skill

Prompt Skill 分三类：

- Scan Core skill catalog：系统内置扫描能力目录。
- Builtin Prompt Skill：每个 agent key 的内置 prompt。
- Custom Prompt Skill：用户自定义 prompt，可全局或 agent-specific 生效。

支持的 agent key：

- `recon`
- `business_logic_recon`
- `analysis`
- `business_logic_analysis`
- `verification`
- `report`

创建智能审计任务时，如果 `use_prompt_skills=true`，后端应生成 prompt runtime snapshot 并写入 `audit_scope`，使任务可追溯当时使用的 prompt 来源。

### 8.3 Prompt 生效顺序

Prompt 合成应遵循：

1. 内置模板。
2. 全局自定义 prompt。
3. agent-specific 自定义 prompt。

详情页和外部工具配置页应能展示 effective prompt 的来源列表，便于审计任务复盘。

## 9. Hermes 运行时需求

### 9.1 Manifest 发现

后端从 `HERMES_AGENTS_BASE_PATH` 读取 role 子目录下的 `agent.toml`，发现可用智能体。

Manifest 必须包含：

- `id`
- `role`
- `image`
- `container_name`
- `enabled`
- `dispatch_timeout_seconds`
- `terminal_cwd`
- `project_mount`
- `artifacts_dir`
- `runtime_home_dir`
- `input_contract.fields`
- `output_contract.fields`
- `healthcheck`

### 9.2 当前角色链路

当前角色固定为：

1. `recon`：仓库结构、入口、依赖、风险面识别。
2. `analysis`：漏洞推理、代码路径分析、候选漏洞生成。
3. `verification`：证据核验、误报过滤、可达性和 PoC 计划。
4. `report`：任务报告和漏洞报告整理。

### 9.3 Handoff Payload

当前 handoff payload 至少包含：

```json
{
  "project_path": "/scan",
  "task_description": "用户创建任务时的描述"
}
```

P1 需要增强 payload：

- 项目元数据。
- target_files。
- exclude_patterns。
- target_vulnerabilities。
- prompt runtime snapshot。
- 静态审计发现摘要。
- 代码浏览索引摘要。
- 上游角色结构化输出。

### 9.4 结构化输出写回

Hermes 输出必须被解析为：

- events。
- findings。
- checkpoints。
- report。
- agent_tree 节点状态。

P1 验收目标：真实 Hermes 输出不再只更新 agent tree summary，而是能写入 `AgentFindingRecord` 和 `AgentEventRecord`。

### 9.5 降级行为

如果 manifests 不存在或为空：

- 当前后端允许降级为 mock completed。
- 产品上必须清楚提示运行时未接入真实 Hermes。
- 生产部署建议把缺失 Hermes manifests 视为预检失败，而不是静默成功。

如果某个 role 执行失败：

- 任务应进入 failed。
- `error_message` 应保留可读失败原因。
- agent tree 对应节点应标记 failed。
- events 应记录 dispatch 错误。

## 10. 静态审计协同需求

### 10.1 当前静态审计能力

Argus 当前已有 Opengrep 静态审计能力：

- Opengrep 规则管理。
- 规则启停、上传、编辑、批量操作。
- 静态任务创建和结果展示。
- 静态发现详情。
- Opengrep runner 镜像和动态 runner 执行。
- CPU / job 自动分配和队列控制。

### 10.2 智能审计协同方式

智能审计应复用静态审计结果：

- 将 Opengrep 高危发现作为智能审计候选输入。
- 对静态审计命中规则补充业务上下文解释。
- 使用 AI 判断静态发现是否可达、是否真实、是否可利用。
- 将智能审计确报结果与静态审计详情页统一到同一漏洞详情模型。

### 10.3 P1 集成需求

创建智能审计任务时允许选择：

- 仅 AI 分析。
- 基于最近一次静态审计结果分析。
- 指定静态审计任务作为输入。
- 仅分析静态审计命中的文件或规则。

任务详情页应展示静态审计输入来源：

- Opengrep task id。
- 规则数量。
- 候选发现数量。
- 进入智能复核的候选数量。

## 11. 安全与隔离需求

### 11.1 项目文件访问

智能审计只能访问被选项目的工作区、目标文件和必要依赖信息。路径展示应使用项目相对路径，避免泄露宿主机无关绝对路径。

### 11.2 API Key 安全

LLM apiKey 属敏感字段：

- 后端保存时必须按现有系统配置加密策略处理。
- 前端展示时不得明文回显完整值。
- 导出的任务报告不得包含 apiKey、customHeaders 敏感值。

### 11.3 容器执行安全

Hermes 和 runner 容器必须满足：

- 通过 manifest 限定角色和容器名。
- 通过 timeout 控制执行时间。
- 对失败 stdout/stderr 做安全截断后展示。
- 不允许用户输入直接拼接为 shell 命令。

### 11.4 漏洞报告安全

导出报告可以包含代码片段，但必须由用户显式选择：

- `include_code_snippets`
- `include_remediation`
- `include_metadata`

默认紧凑报告应避免包含过长代码和敏感配置。

## 12. 性能与资源需求

### 12.1 前端性能

- 智能任务列表轮询间隔默认 5 秒。
- 漏洞列表和事件列表必须分页或限制最大数量。
- 任务详情实时发现最多保留合理窗口，避免长任务导致浏览器内存无限增长。
- 大日志和工具输出应按需展开。

### 12.2 后端性能

- `events/list` 必须支持 `after_sequence` 和 `limit`。
- `findings` 必须支持 `skip` 和 `limit`。
- 报告生成不得阻塞任务状态写入。
- Hermes dispatch 必须有 per-role timeout。

### 12.3 资源控制

当前 Compose 资源基线：

| 服务 | 资源约束 |
| --- | --- |
| backend | `mem_limit: 4g`、`pids_limit: 1024` |
| frontend | `mem_limit: 1g`、`pids_limit: 256` |
| db | `mem_limit: 1g` |
| redis | `mem_limit: 256m` |
| opengrep-runner | `mem_limit: 2g`、`pids_limit: 512` |

Opengrep runner 使用：

- `OPENGREP_SCAN_JOBS=0` 表示自动。
- `OPENGREP_RUNNER_CPU_LIMIT=0` 表示自动。
- `OPENGREP_AVAILABLE_CORES` 可显式声明可用核数。

智能审计后续如引入更多容器，必须复用现有资源感知调度策略，不能固定假设 20 核、32GB 或 GPU。

## 13. 可观测性需求

### 13.1 任务事件

每个关键动作必须写入事件：

- 任务创建。
- 任务启动。
- Hermes role 开始与结束。
- 工具调用开始与结束。
- 漏洞发现。
- 状态更新。
- 报告生成。
- 错误与取消。

### 13.2 Agent 树

Agent 树至少包含 root 和四个 Hermes role 节点。节点字段必须支持：

- `agent_id`
- `agent_name`
- `agent_type`
- `parent_agent_id`
- `status`
- `result_summary`
- `findings_count`
- `verified_findings_count`
- `iterations`
- `tokens_used`
- `tool_calls`
- `duration_ms`

### 13.3 Checkpoint

Checkpoint 应用于：

- 创建后初始化状态。
- role 完成后的结构化状态。
- 发现漏洞后的阶段性快照。
- 任务失败时保存错误上下文。

## 14. 验收标准

### 14.1 创建与预检

- 未配置 LLM 时，智能审计创建按钮不可用，并展示缺失字段。
- 保存并测试 LLM 成功后，可以创建智能审计任务。
- 选择目标文件后，后端任务记录包含 `target_files`。
- 创建任务后，任务出现在 `/tasks/intelligent`。

### 14.2 任务执行

- 启动任务后，状态从 pending 进入 running 或 terminal 状态。
- Hermes manifest 可用时，agent tree 反映四个 role 的执行结果。
- Hermes 不可用时，产品界面明确展示降级或失败原因。
- 失败任务保留 `error_message` 和错误事件。

### 14.3 漏洞结果

- 漏洞列表支持分页、过滤和状态展示。
- 漏洞详情可展示代码、证据、CWE、可达性、PoC 计划和修复建议。
- 标记误报后，误报不计入有效漏洞数。
- 标记确报后，已验证计数和严重程度计数同步更新。

### 14.4 报告导出

- 任务报告支持 markdown、json、pdf 请求。
- 漏洞报告支持 markdown、json、pdf 请求。
- 报告文件名安全化，不包含路径分隔符。
- 紧凑模式下报告不输出过长代码片段。

### 14.5 项目联动

- 项目详情最近任务显示智能审计任务。
- 项目详情可查看最近任务的漏洞详情。
- 仪表盘最近任务支持跳转智能审计详情。
- 静态审计和智能审计的漏洞详情入口一致。

## 15. 实施路线

### 15.1 P0：巩固当前能力

- 保持当前智能审计任务创建、预检、详情、事件、报告能力稳定。
- 修复任务详情、项目详情和统一漏洞详情中的展示问题。
- 保证 agent task API 测试覆盖创建、启动、事件、findings、报告导出。
- 保证前端构建、类型检查和关键 node tests 通过。

### 15.2 P1：真实 Hermes 结构化产出

- 规范 Hermes 输出 JSON schema。
- 将 role 输出写回 events、findings、checkpoints。
- 将 target_files、prompt runtime、静态审计候选输入注入 handoff payload。
- 在任务详情页展示运行时是否真实接入 Hermes。

### 15.3 P1：静态审计与智能审计融合

- 创建智能审计任务时选择静态任务作为输入。
- 对 Opengrep 发现做 AI 复核。
- 将静态规则名、CWE、路径、行号、代码片段映射到智能审计候选。
- 对静态误报和智能误报形成统一状态口径。

### 15.4 P2：数据持久化增强

- 评估将高价值任务数据从快照迁移到数据库。
- 引入任务事件分页索引。
- 支持大型任务归档和历史查询。
- 支持报告文件缓存。

### 15.5 P2：多角色能力增强

- 在现有 Hermes 四角色基础上增加业务逻辑 recon / analysis。
- 增加可配置 role 开关，而不是自动动态拓扑。
- role 增加资源限制、超时和输出 schema 验证。
- 引入角色级质量评分。

## 16. 非目标

当前阶段明确不做：

- 不新增独立编排后端。
- 不重建前端产品体系。
- 不把智能审计任务数据强行迁移到新数据库结构。
- 不引入远程 ECS / EC2 / SSH 调度。
- 不要求 GPU 本地推理。
- 不要求自动生成动态专家容器。
- 不把实验性论文架构描述为当前系统事实。
