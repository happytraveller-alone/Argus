# agent-browser 本地验证页说明

## 阅读定位

- **文档类型**：Explanation（附录）。
- **目标读者**：需要理解这次浏览器调研过程中“最小验证页”思路的开发者。
- **阅读目标**：说明为什么当时需要一个最小 HTML 验证夹具、当前仓库里实际存在什么、以及如果后续要正式化这套夹具还需要补什么。
- **建议前置**：无。
- **本文不覆盖**：并行扫描主架构、Phase 1 实施规范、正式测试体系设计。

## 先说结论

这篇文档是附录，不是 Phase 1 核心实施文档。

它只回答三个问题：

1. 为什么当时需要一个最小浏览器验证页？
2. 当前仓库里到底有没有一个被正式跟踪、可稳定复用的 `verify.html`？
3. 如果后续要把它变成正式测试夹具，还需要补什么？

## 1. 这页当时是用来做什么的

在浏览器页面调研阶段，我们需要一个最小、确定、无业务噪声的页面，来验证下面这条最小链路是否能跑通：

- `agent-browser` CLI 已安装
- 本地页面可打开
- 页面标题可读取
- heading 可读取
- 快照命令能工作

这类页面的价值不在于承载架构内容，而在于：

**把“浏览器自动化工具链本身是否正常”与“业务页面本身是否有问题”分开。**

## 2. 当前仓库里真实存在什么

当前仓库里 **没有** 一个稳定、被 Git 跟踪、可直接复用的正式 `verify.html` 资产。

这意味着：

- 不能把 `verify.html` 当作仓库已经提供的正式测试夹具
- 不能把它当作 Phase 1 文档或实现链路的依赖
- 不能假定仓库已经提供了一个稳定暴露它的本地 HTTP 页面入口

更准确的说法是：

- 这次调研过程中曾经出现过最小验证页草稿
- 但它目前不是本仓库内稳定维护的正式资产

## 3. 当前没有什么

为了避免误解，这里明确列出当前 **没有** 的东西：

- 没有被正式跟踪的 `verify.html`
- 没有专门用于暴露该页面的本地服务脚本
- 没有把该页面纳入自动化测试或开发者工具链文档
- 没有把它接入并行扫描架构正文或 Phase 1 实施规范

因此，这一页只能被理解为：

**一次历史性验证思路的说明，而不是现成可依赖的仓库能力。**

## 4. 为什么仍然保留这篇附录

虽然它不是正式夹具，但这个思路仍然有价值，因为它提醒我们：

- 浏览器工具链排障时，应先验证“工具链通不通”
- 再验证“业务页面内容对不对”

如果一上来就拿复杂页面做验证，就很难快速定位问题到底来自：

- 页面内容
- 本地服务
- 浏览器运行时
- 自动化 CLI 自身配置

## 5. 如果后续要正式化，必须补什么

如果后续要把这类最小验证页升成正式仓库能力，至少要同时补齐下面这些内容：

1. 把页面放到一个明确、被 Git 跟踪的位置。
2. 给出稳定的本地 HTTP 暴露方式，而不是依赖一次性的临时页面。
3. 约定最小 smoke-check 步骤，例如：
   - 打开页面
   - 读取标题
   - 读取 heading
   - 获取快照
   - 关闭浏览器
4. 把它明确标注为“浏览器工具链连通性夹具”，不要和架构正文混用。

## 6. 与 Phase 1 的关系

这篇附录与并行扫描 Phase 1 的关系只有一条：

- 它可以作为“如果后续需要做浏览器自动化自检，可复用最小夹具思路”的历史说明

除此之外：

- 它不是 Phase 1 的依赖项
- 它不参与 control plane / execution plane 边界设计
- 它不参与 queue / workspace / finding store / phase contract 设计

## 7. 一句话总结

这里记录的不是“仓库里已经存在的正式 `verify.html`”，而是：

**一次浏览器能力验证中使用“最小 HTML 夹具”思路的附录说明。**
