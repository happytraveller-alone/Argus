# 漏洞详情页面布局与字体调整设计（2026-03-19）

## 背景与目标
- 用户希望在漏洞详情页中调整为“概要在左、代码在右”，保持主信息优先、代码上下文次级呈现的结构。
- 页面所有文字需要在原字号基础上放大约 30%，并确保放大后的排版不会破坏可读性或响应式体验。
- 需要兼顾可访问性：包括对比度、键盘焦点顺序、触控区域以及小屏幕体验。

## 范围
- `frontend/src/pages/FindingDetail.tsx` 及其子组件（尤其是 `FindingDetailView`、代码面板、属性卡片等）。
- 任何复用漏洞详情布局的模块（例如 Agent/Static 详情汇总）。
- 不涉及其他页面或后端接口。

## 布局方案
1. **顶层 Grid 容器**：
   - 在漏洞详情主体外层设置 `display: grid; grid-template-areas: "main side"; grid-template-columns: minmax(0, 1.02fr) minmax(0, 0.98fr); gap`，使概要区保持在左、代码区保持在右。
   - 将漏洞基本信息、概要、统计卡片等赋值为 `grid-area: main`（左列）。
   - 将代码浏览、上下文、附件等赋值为 `grid-area: side`（右列）。
2. **子区块同步**：
   - 对含两列布局的子组件（例如“处置建议 vs. 证据”）保持“概要/说明在左、辅助/证据在右”的一致语义，避免局部区域再出现反转。
   - DOM 顺序保持 `main`→`side`，桌面与移动端都遵循相同的信息阅读顺序，确保屏幕阅读器与键盘导航先到概要，再到代码。
   - 如果子组件在移动端已经堆叠，继续保持概要在前、代码在后；若某子组件需要特殊顺序，须在文档中加注原因与 QA 步骤。
3. **响应式策略**：
   - 桌面端（>1024px）保持左侧概要、右侧代码的双栏布局。
   - 移动端仍遵循 `"main" "side"` 的自然堆叠，确保最先展示概要信息；如需在特定场景让代码置前，需另行在页面逻辑中显式配置。
   - 检查所有固定宽度元素，必要时改成 `max-width: 100%` + wrap。

## 字体与空间缩放
1. **统一放大约 30%**：
   - 将漏洞详情页内已有文字样式按原始字号乘以 `1.3` 调整，例如 `text-2xl → text-[1.95rem]`、`text-sm → text-[1.1375rem]`。
   - 对仍使用 px/em 的局部样式改为精确 rem 或按比例覆盖，防止局部保留旧字号。
2. **行高/间距**：
   - 正文 `line-height` 提升到 1.6–1.7；标题按比例调整，确保段落不显得拥挤。
   - 卡片、表格的内外边距小幅增加 10–20%，保持视觉节奏但避免因为文字收缩后显得空旷。
3. **特殊内容**：
   - 代码块、日志等等宽字体同样按约 1.3× 放大，优先保证可读性与横向滚动之间的平衡。
   - 徽章、标签等紧凑元素也按约 1.3× 调整字号，并同步微调 padding，防止形态失衡。

## 可访问性与交互
- 颜色对比：放大后重新验证 WCAG 4.5:1，必要时调整变量（尤其是浅色标签）。
- 焦点顺序：保持 DOM 顺序不变；确保键盘 Tab 顺序与 DOM（概要→代码→附加信息）一致，并验证视觉顺序与键盘流一致。
- 触控区域：在字体放大的同时确认按钮/图标的点击区域≥44px，适配 hitSlop。
- 动画/交互：若有过渡效果受 layout 影响，保持 150–300ms 的动画节奏，不阻塞输入。
- 屏幕阅读器：实现后使用 VoiceOver/NVDA 等检查阅读顺序、标题/landmark 宣告顺序是否为 “页面标题 → 概要卡片 → 统计信息 → 代码面板 → 附件/附加信息”；记录任何偏差及 ARIA 调整。

## 实施步骤（概要）
1. 在 `FindingDetail` 页面外层引入新的 grid 容器及样式（可用 CSS Modules 或 Tailwind 自定义类）。
2. 更新 `FindingDetailView` 等子组件，使主要内容与代码区域绑定到 grid areas，并在不改变 DOM 顺序的前提下保持概要左、代码右。
3. 将字体放大逻辑封装到外层容器类，逐个检查使用 px 的元素并替换为 rem。
4. 处理子区块的左右对调与响应式逻辑，确保 breakpoints 下顺序正确。
5. 手动验证：桌面和移动分辨率、深浅主题、键盘导航、代码块滚动。

## 测试计划
- 单元/组件测试：若存在快照，更新 DOM 结构顺序和新增 class。
- 视觉/回归：运行现有 E2E/截图测试（若有），重点关注布局和文本断言。
- 手动验证：
  - 1920px、1280px、1024px、768px、414px 宽度。
  - 中文/英文切换，确保文本长度变化可控。
  - prefers-reduced-motion 设置下无突兀动画。

## 风险与缓解
- **风险**：字体增大导致溢出或折行异常。**缓解**：为表格列添加 `minmax`/wrap、避免硬编码宽度。
- **风险**：键盘顺序与视觉不符。**缓解**：保持 DOM 与视觉同向，并在开发中测试 Tab 顺序。
- **风险**：代码块过宽。**缓解**：设置 `overflow-x: auto`，在宽窗口下提供最大宽度限制。

## 验收标准
- 桌面端视觉与 DOM 都保持“概要在左、代码在右”，键盘/屏幕阅读器顺序为“概要→代码→附加信息”，并在 QA 报告中记录验证结果。
- 所有正文/标题文字尺寸约为原来的 1.3 倍，且无明显布局断裂。
- 代码块、表格等仍可正常滚动/查看，无遮挡或裁剪。
- 可访问性自检通过（对比度、键盘导航、触控区域）。
