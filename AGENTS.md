# AGENTS.md

## 阅读定位

- 文档类型：Reference
- 目标读者：在本仓库内执行任务的主 agent、subagent 和协作开发者
- 阅读目标：在规划、实现、浏览器验证、subagent 协作和 commit 前验证阶段，使用统一且可执行的标准
- 本文不覆盖：业务需求解释、架构背景说明、功能方案讨论

## 总原则

- 主 agent 对最终计划、代码改动、验证结论和 commit 负责。subagent 只负责审阅、挑战、并行分析、局部实现或结果复核，不替代最终判断。
- 不得把未验证内容写成既定事实。无法确认的内容必须标记为“待开发者确认”，并说明影响范围和建议默认值。
- 计划和实施文档不得使用“必要时”“适当处理”“自行判断”这类模糊措辞。
- 所有修改都必须基于 `main` 分支做增量更新；若使用 worktree 或特性分支，也必须从当前 `main` 最新提交切出，不得叠加在未知来源的历史分支上继续开发。

## Skill 与工具约束

- 每个新任务开始时，显式启用 [$pua](/home/xyf/.codex/skills/pua/SKILL.md) 作为执行约束；若任务中调用 subagent，subagent 也必须继承该约束。普通工具调用无需重复附带 skill 链接。
- 允许使用 `agent browser` 和浏览器可视化辅助。涉及页面交互、样式、表单流程、浏览器兼容、截图、录屏、页面回归，或用户明确要求可视化验证时，必须使用。
- 浏览器辅助默认只用于本地环境、测试环境或用户明确授权的目标站点。禁止未授权的生产登录、数据写入、删除、发布、付费等不可逆操作。
- 使用浏览器辅助后，结果中至少记录：访问页面或路由、运行环境、关键操作步骤、期望结果、实际结果，以及截图或日志是否已留存。

## Subagent 规则

- 允许使用 subagent，但只可使用能力高于 `gpt-5.2` 的模型。当前平台优先选择 `gpt-5.4` 或 `gpt-5.3-codex`；不得使用低于或等于 `gpt-5.2` 的模型作为 subagent。
- subagent 适用于：文档审阅、方案挑战、前后端与运维并行分析、测试设计、结果复核，以及写入范围彼此独立的并行实现。
- 多个 subagent 并行时，不得修改同一文件，也不得共享同一未解决写入范围。
- 若当前平台无法确认模型级别，或任务结果强依赖主链路即时推进，subagent 不得成为前置阻塞条件；主 agent 需说明限制后继续完成任务。

## 制定计划时的强制产物

- 形成实施计划前，默认必须从前端、后端、部署运维三个视角审阅相关材料；若用户明确限定单一维度，仍需覆盖该维度，并把其他维度标记为 `N/A` 和原因。
- 审阅对象可以是需求文档、设计文档、现有代码、接口契约、部署配置、测试脚本或页面流程。
- 计划输出必须包含：
  - 背景与目标
  - 已确认事实
  - 前提假设
  - 待开发者确认的问题
  - 影响范围
  - 技术风险
  - 实施步骤草案
  - 验收标准
  - 非目标
- “消除所有前提假设”不是有效目标。正确做法是把假设显式列出，并标记为“已验证 / 待确认 / 可接受默认值”。未确认假设不得写成确定性结论。

## 面向开发者的实施拆解要求

- 计划必须继续拆解到“后续开发者可直接执行”的粒度。每个任务项至少写清：
  - 目标
  - 前置条件
  - 涉及文件或目录
  - 输入与输出
  - 具体修改动作
  - 验证命令或验证方法
  - 完成判定
  - 风险
  - 回滚方案
- 涉及前端改动时，额外写清：受影响页面或组件、交互入口、状态流转、加载态、空态、错误态、禁用态、响应式影响和浏览器验证方式。
- 涉及后端改动时，额外写清：接口或任务入口、数据模型或存储影响、并发、取消、重试、鉴权边界，以及回归验证方式。
- 涉及部署或运维改动时，额外写清：compose、镜像、环境变量、启动顺序、健康检查、回滚路径和联调影响。
- 只有在仓库、现有文档、测试结果和浏览器验证都无法给出答案时，才允许向开发者确认。提问前必须先说明已检查证据、阻塞范围和建议默认值。

## Commit 前验证门禁

- 每次 `commit` 前，必须执行与本次改动范围匹配的前后端验证，并在汇报中逐条记录命令和结果。
- 修改 `frontend/` 或前端构建配置时，在 `frontend/` 目录执行：
  - `pnpm lint`
  - `pnpm type-check`
  - `pnpm test:node`
  - `pnpm build`
- 修改 `backend/` 时，在 `backend/` 目录执行：
  - `cargo test`
  - `cargo build --bin backend-rust`
- 同时修改前后端时，前后端命令都必须执行。
- 修改 `docker-compose.yml`、`docker/`、镜像、环境变量、启动脚本或服务依赖关系时，额外执行：
  - `docker compose config`
  - 若改动影响服务启动或联调，再执行 `docker compose up --build -d`
  - `docker compose ps`
  - 验证 `backend` 的 `/health` 通过，并确认前端页面可访问
- `backend_old/` 仅在本次改动触及该目录时才纳入验证范围；未改动时，不属于当前后端门禁。
- 通过标准：
  - 所有必跑命令退出码为 `0`
  - 不得引入新的 `error` 或 `warning`
  - 若存在历史遗留问题或第三方 warning，必须在汇报中注明来源、影响范围和未处理原因
- 因环境限制无法完成验证时，不得声称“已通过”或直接执行 `commit`。必须先报告未执行项、阻塞原因、潜在风险和补跑建议，并等待开发者确认。
